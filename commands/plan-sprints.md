## Задача: $ARGUMENTS

## Pipeline

Ты — координатор декомпозиции. Выполняй pipeline автономно, без вопросов.

### Шаг 0 — Анализ входа

Прочитай указанную спеку/план. Определи:
- Сколько спринтов нужно декомпозировать
- Какие спринты уже готовы (есть в `docs/product/`)
- Какие нужно создать

Количество агентов = количество спринтов для декомпозиции. НЕ фиксировано.

### Этап 1 — PM (продуктовые ТЗ)

Запусти PM параллельно — по одному на каждый спринт:
- subagent_type: "pm"
- model: "opus"
- Каждый PM читает спеку фазы + свой спринт
- PM пишет продуктовое ТЗ в `docs/product/SPRINT-X.Y-name.md`

PM продумывает ПРОДУКТ (не техническую обёртку):
- Кто пользователь, какой сценарий, какой UX
- Формат данных (колонки, числа, цвета, порядок)
- Информационная архитектура (что видит первым)
- Разбивка на 5-8 мини-задач (каждая = конкретный видимый результат)
- Пишет на русском

Жди завершения ВСЕХ PM.

### Этап 2 — Architect (техпланы)

Запусти Architect параллельно — по одному на каждую PM-спеку:
- subagent_type: "architect"
- model: "opus"
- Каждый Architect читает PM-спеку + техническую спеку фазы + код проекта
- Architect обогащает каждую мини-задачу: точные файлы, сигнатуры функций, SQL, API контракты
- Пишет результат прямо в PM-файл (секция "Технический план" для каждой мини-задачи)

Жди завершения ВСЕХ Architect.

### Этап 3 — Planner (валидация)

Запусти Planner параллельно — по одному на каждый спринт:
- subagent_type: "planner"
- model: "opus"
- Planner проверяет автономно: полнота, консистентность, реализуемость
- Вердикт: APPROVED или NEEDS REVISION (с конкретными замечаниями)

Если NEEDS REVISION:
- Передай замечания Architect -> Architect правит -> Planner перепроверяет
- Максимум 3 цикла ревизии
- Если после 3 циклов не APPROVED — пометь спринт как [needs attention]

Жди APPROVED на все спринты.

### Этап 4 — GitHub Issues

После всех APPROVED:
- Для каждой мини-задачи из каждого спринта создай GitHub sub-issue
- Формат title: "Sprint X.Y: [название мини-задачи]"
- Body: описание задачи из PM-спеки + технические детали из Architect
- Labels: sprint-X.Y, phase-N
- Привяжи к parent issue спринта (если есть)

### Этап 5 — Коммит + отчёт

Закоммить все `docs/product/SPRINT-*.md`.

Финальное сообщение:
```
Декомпозиция готова:
- [список созданных SPRINT-*.md файлов]
- [количество] GitHub issues создано
- Всё APPROVED

Следующий шаг: создай workspace'ы в Conductor для реализации (Cmd+Shift+N -> Issues)
```
